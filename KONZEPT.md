## 1. PROJEKTVERSTÄNDNIS

1. **Auftraggeber** ist die FBG Schlaubetal, eine anerkannte Forstbetriebsgemeinschaft in Brandenburg (Oder-Spree / Schlaubetal), Sitz: Siedlung 18, 15848 Ragow-Merz.
2. **Ausgangslage**: Die FBG hat aktuell keine eigenständige, belastbare digitale Präsenz. Das ist im Vergleich zu ähnlichen Organisationen ein Defizit — und gleichzeitig eine Chance.
3. **Primärziel** der Website ist nicht Leadgenerierung oder Verkauf, sondern **seriöse Außendarstellung** und **klare Information** für bestehende und potenzielle Mitglieder.
4. **Zielgruppen** sind vorrangig private Waldbesitzer in der Region, bestehende Mitglieder, potenzielle Neumitglieder — sekundär Behörden, Dienstleister und Akteure aus Forst/Naturschutz.
5. **Tonalität**: Die Seite soll Kompetenz, Verlässlichkeit und regionale Verankerung kommunizieren — sachlich, freundlich, hochwertig, aber nicht überstilisiert.
6. **Differenzierung**: Vergleichbare FBG-Websites sind häufig funktional, aber visuell veraltet und informationsarchitektonisch schwach. Die FBG Schlaubetal kann sich durch Klarheit, Nutzerführung und professionelle Gestaltung deutlich abheben.
7. **Umfang**: Kleine bis mittlere Website mit überschaubarem Pflegeaufwand — kein Portal, kein Shop, keine komplexe Web-App.
8. **Pflege**: Die Redaktion wird vermutlich durch nicht-technische Personen erfolgen. Pflegefreundlichkeit ist daher ein harter Designfaktor.
9. **Regionaler Bezug**: Das Schlaubetal als Landschaftsraum soll spürbar sein — nicht als Deko, sondern als Identitätsmerkmal.
10. **Langfristperspektive**: Die Website soll erweiterbar sein (News, Downloads, Mitgliederbereich), aber V1 muss schlank und solide funktionieren.

## 6. DESIGNRICHTUNG

### Stimmung
Ruhig, kompetent, naturverbunden, regional. Wie ein gut geführter Forstbetrieb: professionell, aber nicht steril. Vertrauenswürdig, nicht werblich.

### Farbwelt

| Rolle | Farbe | Hex (Vorschlag) | Anmerkung |
|---|---|---|---|
| Primär | Waldgrün (dunkel) | `#2D5016` | Ankert die Seite im Forst-Kontext |
| Primär hell | Waldgrün (hell) | `#4A7C2E` | Hover, Akzente |
| Sekundär | Warmes Sandbeige | `#F5F0E8` | Hintergründe, Wärme |
| Akzent | Erdbraun | `#8B6914` | Sparsam, für Highlights |
| Text | Anthrazit | `#2C2C2C` | Lesbarkeit |
| Hintergrund | Warmweiß | `#FAFAF7` | Nicht klinisch weiß |
| Hintergrund alt | Hellbeige | `#F0EBE0` | Für Section-Wechsel |

### Typografie
- **Headlines**: Serifenschrift — vermittelt Seriosität und Tradition. Vorschlag: **Merriweather** oder **Lora** (Google Fonts, kostenlos).
- **Body**: Serifenlose Schrift — klar, modern, gut lesbar. Vorschlag: **Inter** oder **Source Sans 3**.
- **Größen**: Großzügig, gut lesbar auch auf Mobilgeräten. Mindestens 18px Body-Text.

### Bildsprache
- Authentische Fotos: Mischwald, Brandenburger Landschaft, Schlaubetal
- Warme, natürliche Lichtstimmung (kein HDR-Overkill)
- Menschen im Wald: Förster, Waldbesitzer, Gemeinschaft (falls vorhanden)
- Keine generischen Stockfotos von „glücklichen Geschäftsleuten"
- Falls keine eigenen Fotos vorhanden: hochwertige, regionsnahe Waldbilder (z.B. von Unsplash, lizenzfrei)
- Dezente Icons für Leistungen (Linienicons, nicht bunt/verspielt)

### Layout-Prinzipien
- **Großzügiger Weißraum** — Seite atmet, nichts ist gedrängt
- **Klare Hierarchie** — Headline → Subline → Text → CTA
- **Maximalbreite** für Text: ~720px (Lesbarkeit)
- **Sections mit wechselnden Hintergründen** — Warmweiß / Hellbeige im Wechsel
- **Keine Slider / Karussells** — stattdessen statische, starke Bilder
- **Karten-Layout** für Leistungsübersichten
- **Sticky Navigation** — dezent, nicht dominant

### Mobile UX
- Mobile-First-Denken bei der Implementierung
- Hamburger-Menü mit vollem Overlay
- Große Touch-Targets (min. 44px)
- Telefon- und E-Mail-Links klickbar
- Karte einbettbar und scrollbar
- Keine horizontalen Scrolls

### Was vermieden werden soll
- ❌ Grelle Farbverläufe, Neon-Akzente
- ❌ Parallax-Effekte, aufwendige Animationen
- ❌ Dunkle Themes / Dark Mode (passt nicht zur Zielgruppe)
- ❌ Generische 3-Spalten-Feature-Grids mit bunten Icons
- ❌ „Startup-Feeling" in jeder Form
- ❌ Stock-Bilder von Handshakes oder Laptops
- ❌ Überladene Footer mit Social-Media-Icons (sofern keine Kanäle vorhanden)

## 7. TECH-STACK-EMPFEHLUNG

### Bewertung der Optionen

#### Framework

| Kriterium | Astro | Next.js | Statisches HTML/CSS/JS |
|---|---|---|---|
| Performance | ⭐⭐⭐⭐⭐ (0 JS default) | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| Wartbarkeit | ⭐⭐⭐⭐ (Komponenten) | ⭐⭐⭐⭐ | ⭐⭐ (kein Komponentensystem) |
| Lernkurve | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| CMS-Integration | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐ |
| Passend für Projekt | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ (Overkill) | ⭐⭐⭐ (nicht skalierbar) |

**→ Empfehlung: Next.js mit App Router** — allerdings mit einer klaren Einschränkung:

Für eine rein statische, kleine Website wäre **Astro** die technisch passendste Wahl: zero JavaScript by default, hervorragende Performance, Markdown-Content. **Aber**: Da du bereits im Next.js/Vercel-Ökosystem arbeitest und die Frontend-Design-Skills darauf ausgelegt sind, ist **Next.js mit statischem Export** die pragmatisch beste Wahl. Es liefert:
- Server Components (kein unnötiges JS an den Client)
- Komponentenbasierte Architektur
- Einfaches Deployment auf Vercel
- Erweiterbar, falls später dynamische Features nötig werden
- Bildoptimierung mit `next/image`

#### Styling

**→ Empfehlung: Tailwind CSS**
- Schnelle Iteration
- Konsistente Design-Tokens (Farben, Abstände, Schriften)
- Kein CSS-Bloat
- Gut wartbar bei kleinen bis mittleren Projekten

#### CMS

| Option | Bewertung |
|---|---|
| Kein CMS (Markdown/Code) | Gut für V1, nicht pflegefreundlich für Nicht-Techniker |
| Decap CMS (ehemals Netlify CMS) | Git-basiert, kostenlos, einfach — aber UX ist begrenzt |
| Sanity | Sehr flexibel, kostenloser Tier — aber Setup-Aufwand |
| Tina CMS | Git-basiert, visual editing — gute Option |

**→ Empfehlung für V1: Kein CMS.** Inhalte direkt im Code (als Komponenten/Daten). Gründe:
- Die Inhalte ändern sich selten (kein Blog, keine täglichen News in V1)
- Weniger Komplexität = weniger Fehlerquellen
- Schneller Go-Live
- **Für V2** (wenn News/Aktuelles dazukommen): Evaluation von Sanity oder Tina CMS

#### Deployment

**→ Empfehlung: Vercel**
- Zero-Config für Next.js
- Automatische Deployments bei Git-Push
- Preview-Deployments für Reviews
- SSL, CDN, Performance-Optimierung inklusive
- Kostenloser Tier reicht für dieses Projekt

#### Formularlösung

**→ Empfehlung: Server Action + E-Mail-Versand** oder alternativ ein einfacher externer Dienst:
- **Option A**: Next.js Server Action → E-Mail via Resend (kostenloser Tier: 100 E-Mails/Tag)
- **Option B**: Formspree oder Formspark (hosted, DSGVO-kompatibel prüfen)
- **Option C**: Einfaches `mailto:`-Link als Fallback

**→ Empfehlung: Option A** (Server Action + Resend) — volle Kontrolle, kein externer Dienst nötig, DSGVO-konform konfigurierbar.

#### Zusammenfassung Tech-Stack

| Komponente | Empfehlung |
|---|---|
| Framework | Next.js 15+ (App Router, Server Components) |
| Styling | Tailwind CSS |
| Typografie | Google Fonts (Merriweather + Inter) |
| Icons | Lucide Icons |
| Bilder | next/image + WebP |
| CMS | Keins in V1, Evaluation für V2 |
| Formulare | Server Action + Resend |
| Hosting | Vercel |
| Versionskontrolle | Git + GitHub |

1. **Next.js-Projekt aufsetzen** mit App Router, Tailwind CSS, Google Fonts (Merriweather + Inter), und der definierten Farbpalette.

→ Das ergibt einen ersten sichtbaren, klickbaren Prototyp, der als Diskussionsgrundlage mit der FBG dienen kann.
